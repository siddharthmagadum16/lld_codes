## ğŸ¨ Design Pattern Application

### 1. State Pattern

```
Context: AtmMachineService
States: ReadyState, CardInsertedState, etc.
Behavior: Changes based on current state
```

### 2. Strategy Pattern

```
Context: DispatchCashState
Strategies: GreedyWithdrawalStrategy, OptimalWithdrawalStrategy (future)
Behavior: Different withdrawal algorithms
```

### 3. Singleton Pattern

```
Instances: AtmMachineService, CashDao
Purpose: Single instance throughout application
Note: Dependencies still injected for testability
```

### 4. Dependency Injection

```
Pattern: Constructor Injection
Benefits: Loose coupling, testability, flexibility
Applied: States receive dependencies via constructor
```

## ğŸ¯ Key Architecture Principles

1. **Separation of Concerns**
   - States manage transitions
   - Context manages data
   - Service coordinates
   - Strategies handle algorithms
   - Validators handle validation

2. **Dependency Inversion**
   - High-level modules depend on interfaces
   - Low-level modules implement interfaces
   - Dependencies injected, not created

3. **Single Responsibility**
   - Each class has one reason to change
   - Clear boundaries between components

4. **Open/Closed**
   - Open for extension (new strategies, validators)
   - Closed for modification (existing code unchanged)

5. **Testability**
   - All dependencies can be mocked
   - Each component testable in isolation

---

This architecture ensures maintainability, extensibility, and follows all SOLID principles.
